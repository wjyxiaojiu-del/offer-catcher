# Offer 捕手 — 优化路线图

> 基于 v1.1 代码审计，按优先级分档给出可执行的改进建议。

---

## P0 — 立刻做（性价比最高，影响最大）

### 1. 数据持久化从浏览器存储迁移到数据库

**问题**：当前简历、投递记录等核心数据大量依赖 `localStorage`/`sessionStorage`（10 个文件 42 处）。用户换浏览器/清缓存/换设备后数据全丢。

**影响**：
- 用户刷新页面后简历需要重新上传
- 投递记录在无痕模式下无法留存
- 多设备使用体验极差

**方案**：
- 简历上传后写入 `Resume` 表，返回 `resumeId`，URL 携带 `?resumeId=xxx`
- 投递记录完全走 `Application` 表（目前已经有表，但页面仍从 localStorage 读）
- 面试学习进度已在 DB，没问题
- 移除 `applications/page.tsx` 中的 `localStorage.getItem("applications")` 回退逻辑

**预期收益**：数据真正持久化，用户体验从"玩具"升级为"工具"。

---

### 2. 面试刷题增加「上一题/下一题」导航

**问题**：面试题详情页 (`/interview/questions/[id]`) 只能返回列表，不能连续刷题。每看完一题要回到列表再点下一题，体验极差。

**方案**：
- 在题目详情页底部添加「上一题」「下一题」按钮
- 根据当前筛选条件（模块/难度/状态）预加载相邻题目的 ID
- 支持键盘快捷键 `←`/`→` 切换

**预期收益**：刷题效率提升 3-5 倍，用户更愿意持续使用。

---

### 3. 引入 SWR / React Query 统一数据获取

**问题**：当前所有页面都是手写 `useEffect + fetch`，没有缓存、没有乐观更新、没有错误重试。同一个 API（如 `/api/interview/stats`）在不同页面被重复请求。

**影响**：
- 页面切换时数据闪烁
- 网络抖动时直接白屏或显示 0
- 代码重复度高

**方案**：
- 安装 `swr` 或 `@tanstack/react-query`
- 封装通用 fetcher，自动附加 `Authorization` token
- 高频数据（题库列表、学习进度）加 `dedupingInterval` 和 `revalidateOnFocus`

```tsx
// 示例：封装后
const { data: stats } = useSWR('/api/interview/stats', fetcher)
```

**预期收益**：代码减少 30%，用户体验更流畅，为 PWA 离线能力打基础。

---

### 4. 岗位库扩容到 200+（或接入外部数据源）

**问题**：内置岗位只有 25 条，用户上传简历后匹配结果少，演示效果差。

**方案（二选一）**：

**方案 A — 批量导入脚本**：
- 写一个简单的 CSV/JSON 导入脚本，支持从外部数据源批量写入 `Job` 表
- 在 Admin 页面提供「导入岗位」功能

**方案 B — 接入公开 API**：
- 接入 51job / 拉勾 / Boss 直聘 的公开职位搜索 API（仅搜索，不抓取）
- 用户搜索时实时拉取并写入本地 DB 做缓存

**短期最低成本方案**：
- 手工整理 100-200 条真实岗位数据到 `data/jobs.ts`，按行业分类（前端/后端/AI/产品/数据/测试/运维/安全/移动端/设计）
- 每条岗位包含真实的技能要求、薪资范围、地点

**预期收益**：匹配功能从"演示"变成"真正能帮用户发现岗位"。

## P1 — 中期做（体验提升明显）

### 5. 暗色模式（Dark Mode）

**问题**：没有暗色模式。现代应用标配，夜间使用刺眼。

**方案**：
- Tailwind 已支持暗色：`darkMode: 'class'` 或 `'media'`
- 在 `layout.tsx` 根节点加 `dark:` 前缀的样式
- 在 Navbar 加切换按钮，偏好写入 `localStorage`（用户偏好，非核心数据）
- 面试刷题页面优先适配（用户可能长时间盯着屏幕）

**预期收益**：视觉体验现代化，符合用户预期。

---

### 6. 模拟面试 SSE 解析改用标准 EventSource

**问题**：当前 SSE 解析是手写字符串分割：`text.split('\n').filter(line => line.startsWith('data: '))`，对断包、粘包、JSON 解析异常的处理不够健壮。

**方案**：
- 前端改用标准 `EventSource` API 或 `fetch-event-source` 库
- 后端保持 SSE 格式不变
- 增加连接断开自动重连、心跳保活

```tsx
// 改进后
import { fetchEventSource } from '@microsoft/fetch-event-source'

fetchEventSource('/api/interview/mock/chat', {
  method: 'POST',
  body: JSON.stringify({...}),
  onmessage(msg) {
    const data = JSON.parse(msg.data)
    // 处理各类型事件
  },
  onerror(err) {
    // 自动重连
  }
})
```

**预期收益**：面试对话稳定性提升，弱网环境下也能正常使用。

---

### 7. 增加数据导出功能

**问题**：用户的匹配结果、面试复盘报告、投递记录都无法导出，数据被锁定在系统里。

**方案**：
- **匹配报告导出**：生成 Markdown/PDF 格式的匹配分析报告
- **面试复盘导出**：支持导出为 Markdown（含分数、维度评价、改进建议）
- **投递记录导出**：CSV 格式（岗位、公司、状态、投递时间）
- 纯前端实现：用 `jsPDF` 或简单 `window.print()` + CSS 媒体查询

**预期收益**：用户感知到"数据属于自己"，信任度提升。

---

### 8. 面试题库管理后台

**问题**：当前面试题只能通过 `seed-interview.ts` 脚本导入，普通用户无法添加自己的题目。

**方案**：
- 新增 `/admin/questions` 页面（简单表单即可）
- 支持 CRUD：添加题目、编辑答案、修改模块和难度
- 无需复杂权限，token 鉴权即可

**预期收益**：社区可以共建题库，题目数量快速增长。

---

### 9. Agent ReAct 循环支持多轮迭代

**问题**：当前 TODO 中 `#9: B3 - ReAct 循环升级 (最多 3 轮)` 标记为部分完成。Agent 只能执行一轮 Plan → Execute → Respond，无法根据执行结果动态调整计划。

**场景**：
- 用户说"帮我投递匹配度最高的 3 个岗位"
- Agent 匹配后发现最高的才 55 分，应该自动追问"匹配度较低，是否降低标准或先优化简历？"

**方案**：
- 在 `orchestrator.ts` 的 `runAgent` 中增加循环逻辑
- 每次执行后检查是否有任务失败/结果异常
- 如果结果不理想，用 LLM 判断是否需要重新规划（最多 3 轮）

**预期收益**：Agent 从"执行命令"升级为"有判断力的助手"。

## P2 — 长期做（锦上添花）

### 10. PWA 离线刷题能力

**问题**：面试刷题场景很适合在地铁/等餐等碎片化时间使用，但当前必须联网。

**方案**：
- 添加 `manifest.json` 和 Service Worker
- 用 Workbox 缓存题库静态资源和已加载的题目数据
- 学习进度在联网时同步到服务端

**预期收益**：使用场景扩展，用户粘性提升。

---

### 11. 多 LLM  Provider 支持

**问题**：当前 `lib/ai.ts` 硬编码小米 MiMo。如果用户没有 MiMo Key，只能降级到规则引擎。

**方案**：
- 抽象 `LLMProvider` 接口：
  ```ts
  interface LLMProvider {
    chat(messages: Message[], options?: ChatOptions): Promise<string>
    stream?(messages: Message[]): AsyncIterable<string>
  }
  ```
- 实现 `OpenAIProvider`、`DeepSeekProvider`、`OllamaProvider`（本地模型）
- 环境变量 `LLM_PROVIDER=openai` 切换

**预期收益**：用户可以用自己的 API Key（OpenAI、DeepSeek、通义千问等），甚至完全离线使用 Ollama。

---

### 12. 全局搜索

**问题**：岗位、面试题、聊天记录分散在各页面，没有统一入口搜索。

**方案**：
- 在 Navbar 添加搜索框（类似 Command+K）
- 使用 `cmdk` 库实现快捷搜索面板
- 搜索范围：岗位标题/公司、面试题、最近聊天记录

**预期收益**：信息检索效率提升，尤其题库大了之后。

---

### 13. 接入真实投递渠道（可选，不商用）

**问题**：批量投递明确是"模拟"，用户可能会问"能不能真的投？"

**方案（研究性质，不违反平台 ToS）**：
- 保留现有模拟投递作为"沙盒演练"
- 在 BOSS 直聘页面增加"投递记录同步"功能：将模拟投递的岗位列表一键导入到 BOSS 真实搜索关键词中
- 或者生成一个"今日投递清单"Markdown，用户自己复制到各平台投递

**预期收益**：从"模拟"到"辅助真实行动"的桥梁。

---

## 建议实施顺序

```
第 1 周：P0-1（数据持久化） + P0-2（题目导航）
第 2 周：P0-3（SWR） + P0-4（岗位扩容）
第 3 周：P1-5（暗色模式） + P1-6（SSE 优化）
第 4 周：P1-7（数据导出） + P1-8（题库管理）
第 5-6 周：P1-9（ReAct 多轮） + P2-10（PWA）
第 7-8 周：P2-11（多 LLM） + P2-12（全局搜索）
```
